# Create Technical Plans for Common Functionality

Create detailed technical planning directories for each piece of common functionality identified in the cross-cutting concerns analysis.

## Usage

```
/create-planning-directories-for-common-functionality
```

No parameters required. This command processes the output from `/review-technical-plans` to create planning directories for shared components.

## Prerequisites

This command must be run AFTER `/review-technical-plans` has completed and generated `./docs/plan/common.md`.

## What This Command Does

This command transforms the cross-cutting concerns analysis into actionable technical planning directories. For each piece of common functionality identified in `./docs/plan/common.md`, it creates:

1. **A containing directory** at `./docs/plan/common/{key}/`
2. **A detailed description file** at `./docs/plan/common/{key}/description.md`

The description files serve as comprehensive technical briefs that provide all the context a developer needs to:
- Create a full functional specification
- Design the technical implementation
- Understand integration points with existing features
- Locate relevant reference code in the legacy codebase

**What This Command Does NOT Do:**
- Generate actual functional specifications (those come later)
- Generate technical implementation plans (those come later)
- Write implementation code

## Input

### Source Document

`./docs/plan/common.md` - The cross-cutting concerns report generated by `/review-technical-plans`

This document contains:
- **Critical Extractions**: Must-have shared components
- **High Priority Extractions**: Frequently-used components
- **Medium Priority Extractions**: Moderately-duplicated components
- **Low Priority Extractions**: Minor standardization opportunities
- **Common Patterns Catalog**: Categorized patterns across all technical plans

**IMPORTANT**: Each common functionality item MUST have a **"key"** field that serves as the unique identifier and directory name. This key should be in kebab-case format (e.g., `base-repository`, `audit-logging-service`).

## Output

### Directory Structure

For each common component identified, create:

```
./docs/plan/common/
├── {component-key-1}/
│   └── description.md
├── {component-key-2}/
│   └── description.md
└── {component-key-N}/
    └── description.md
```

### Naming Conventions

**CRITICAL**: Directory names are determined by the **"key"** field from each common functionality item in `common.md`.

The key should follow these conventions:
- **Lowercase with hyphens**: `base-repository` not `BaseRepository`
- **Descriptive and specific**: `audit-logging-service` not `audit`
- **Technology-agnostic**: `permission-guard` not `spring-security-guard`
- **Singular form**: `validation-utility` not `validation-utilities`

**DO NOT** generate or transform the key yourself - use the exact key value from `common.md`.

### Priority Mapping

Create directories for components with these priorities from common.md:
- ✅ **Critical**: Must create
- ✅ **High**: Must create
- ✅ **Medium**: Must create
- ⚠️ **Low**: Create only if it represents a reusable component (not just a documentation pattern)

## Execution Instructions

### Phase 1: Read and Parse

1. **Read common.md**:
   ```
   ./docs/plan/common.md
   ```

2. **Extract all components** from these sections:
   - Critical Extractions
   - High Priority Extractions
   - Medium Priority Extractions
   - Low Priority Extractions (only actual components, not just patterns)

3. **For each component, extract**:
   - **Key** (REQUIRED - the unique identifier that will be used as the directory name)
   - Component name (the human-readable display name)
   - Type (Service/Repository/Model/etc.)
   - Priority level
   - Which plans use it
   - How it's currently implemented (variations)
   - Recommended extraction design

   **VALIDATION**: If any component is missing a "key" field, STOP and report the error. Do not proceed without keys.

### Phase 2: Create Directory Structure

1. **Ensure base directory exists**:
   ```bash
   mkdir -p ./docs/plan/common
   ```

2. **For each component**:
   - Use the **key** field as the directory name (DO NOT transform or generate it)
   - Check if directory `./docs/plan/common/{key}/` already exists
   - **IF DIRECTORY EXISTS**:
     - ⚠️ **SKIP** this component entirely
     - Log: "Skipped {key}: directory already exists"
     - Do NOT overwrite, modify, or delete existing content
     - Continue to next component
   - **IF DIRECTORY DOES NOT EXIST**:
     - Create directory `./docs/plan/common/{key}/`
     - Verify creation succeeded
     - Log: "Created {key}"
     - Proceed to Phase 3 for this component

   **CRITICAL**: NEVER overwrite existing directories. Existing planning work must be preserved.

### Phase 3: Generate Description Files

**IMPORTANT**: Only generate description files for components whose directories were **newly created** in Phase 2. Skip components that already had existing directories.

For each **newly created** component directory:

1. **Gather all relevant information from common.md**:
   - Where it appears
   - How it's used
   - Current implementation details
   - Recommended design
   - Benefits
   - Implementation notes

2. **Enrich with legacy code references**:
   - Search technical plans for the component mentions
   - Extract file paths and locations from call trees
   - Identify database dependencies from database-dependencies.json files
   - Find relevant code in legacy codebase

3. **Design target implementation**:
   - Draft interface/class signatures based on recommendations in common.md
   - Add TypeScript/Java code examples
   - Define key data models
   - Specify integration points

4. **Add implementation guidance**:
   - Technical challenges and mitigations
   - Testing strategy
   - Security and performance considerations
   - Do's and don'ts

5. **Cross-reference with architecture**:
   - Link to relevant target architecture documents
   - Reference applicable design patterns
   - Cite framework documentation

6. **Write description.md**:
   - Follow the template structure exactly
   - Be comprehensive but concise
   - Include all actionable information
   - Provide clear next steps

### Phase 4: Validation

After creating all directories and files:

1. **Verify completeness**:
   - [ ] All Critical priority components have directories (either pre-existing or newly created)
   - [ ] All High priority components have directories (either pre-existing or newly created)
   - [ ] All Medium priority components have directories (either pre-existing or newly created)
   - [ ] Each **newly created** directory has a description.md file
   - [ ] No empty or stub files in newly created directories
   - [ ] Pre-existing directories were not modified

2. **Verify quality** (for newly created files only):
   - [ ] Each new description includes legacy code references
   - [ ] Each new description includes code examples
   - [ ] Each new description lists all technical plans that use the component
   - [ ] Each new description provides clear implementation guidance

3. **Generate summary report**:
   - Count of directories **newly created**
   - Count of directories **skipped** (already existed)
   - Total count of all common functionality directories
   - List of newly created component keys
   - List of skipped component keys (already existed)
   - Breakdown by priority (new vs existing vs total)
   - Breakdown by type/category

- Always check if `./docs/plan/common/{key}/` exists before creating
- If it exists, skip it completely and log the skip
- Preserving existing work is more important than completeness
- Report both created and skipped directories in the summary

- Every component in common.md MUST have a "key" field
- Use this key exactly as-is for directory names (no transformation)
- Do not generate or infer keys - they must be explicitly present
- If a key is missing, report an error and stop processing

- Track which directories were skipped (already existed)
