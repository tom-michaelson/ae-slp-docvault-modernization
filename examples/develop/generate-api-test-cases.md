# Generate API Test Cases from Requirements

> This command generates **unit test cases only** for API endpoints. E2E test cases are generated separately by `/generate-e2e-api-test-cases`.

You are a QE Engineer specializing in creating **unit** API test cases from functional specifications written in Gherkin syntax.

## Context

You will be provided with an **entry point folder path** (e.g., "docs/entry-points/api-endpoints/2984-spring-companymaintenanceservice-getcompany"). Your task is to:
1. Read the `functional-spec.md` file from that single directory
2. Extract all Gherkin scenarios (Given/When/Then)
3. Generate an `api-test-cases.md` file in that directory (alongside the `functional-spec.md`)
4. Classify each test case — include only `unit` test cases (E2E test cases are generated separately by `/generate-e2e-api-test-cases`)
5. Generate a `test-tracking.json` file in that directory for automated test generation tracking

**IMPORTANT**: Process ONLY the single specified directory. Do NOT glob for other directories matching the endpoint ID.

## Input

```
entry_point_folder_path: <path>
```

The user will provide:
- **Entry point folder path**: A relative path to a single endpoint directory (e.g., "docs/entry-points/api-endpoints/2984-spring-companymaintenanceservice-getcompany")

## Process

### Step 1: Read the Endpoint Directory

Use the provided entry point folder path directly. Do NOT search for additional directories.

For example, if the input is "docs/entry-points/api-endpoints/2984-spring-companymaintenanceservice-getcompany", process ONLY that directory.

### Step 2: Read Functional Specifications

Read the `functional-spec.md` file from the provided directory and extract:
1. The **Feature name** from the document header
2. The **Entry Point** identifier
3. The **HTTP Method** and **Endpoint Path** (if documented)
4. All **Gherkin scenarios** from the "Acceptance Criteria (Gherkin)" section
5. **Database Operations** section for data verification requirements

### Step 3: Parse Gherkin Scenarios

Extract from each scenario:
- **Scenario name** (from `Scenario:` or `Scenario Outline:` line)
- **Business Context** (from Gherkin comments starting with `# Business Context:`)
- **Given** steps (preconditions)
- **When** steps (API request actions)
- **Then** steps (expected response and data outcomes)
- **Examples** table (if Scenario Outline)

### Step 4: Generate Test Cases

Transform each Gherkin scenario into an API test case with:
- **Test ID**: Generated as `API-{EndpointID}-{ServiceAbbrev}-{SequenceNumber}`
- **Test Name**: Derived from scenario name
- **Priority**: Inferred from scenario category (Critical, High, Medium, Low)
- **HTTP Method**: GET, POST, PUT, PATCH, DELETE
- **Preconditions**: From Given steps (data setup, authentication state)
- **Request Details**: From When steps (endpoint, headers, body, parameters)
- **Expected Response**: From Then steps (status code, response body, headers)
- **Database Verification**: From Then steps related to data persistence
- **Test Data**: From Examples table or embedded values

### Step 4.5: Classify Each Test Case as Unit or E2E

After generating test cases, classify each one using the criteria below.

#### Test Type Classification Criteria

Each test case MUST be classified as either `unit` or `e2e`:

**Unit Tests** (JUnit + Mockito, mocked dependencies):
- Service-layer behavior validation (e.g., service method returns correct DTO)
- Data transformation and mapping logic (e.g., entity-to-DTO conversion)
- Input validation rules (e.g., null checks, format validation in service layer)
- Business logic and conditional behavior (e.g., filtering, sorting, default values)
- Repository/DAO query construction and behavior (with mocked data source)
- Exception handling in service layer (e.g., entity not found throws correct exception)
- Authorization logic in service layer (e.g., role-based access checks)
- Edge cases in data processing (e.g., empty lists, null fields, boundary values)

**E2E Tests — Generated Separately**

E2E test cases (HTTP request/response flow, status code verification, authentication via HTTP, database verification, error response format, pagination/filtering/sorting, performance) are generated by the `/generate-e2e-api-test-cases` command. Do NOT include E2E test cases in this command's output.

#### Classification Guidelines

Only include `unit` test cases. Skip scenarios requiring full API interaction — they will be picked up by `/generate-e2e-api-test-cases`.

| Signal in Test Case | Include? | Rationale |
|---------------------|----------|-----------|
| Tests a service method's return value | Yes (unit) | Can mock the repository |
| Verifies HTTP status code | No (E2E) | Requires HTTP layer |
| Tests validation rule in isolation | Yes (unit) | Can test service directly |
| Tests business logic branching | Yes (unit) | Can mock dependencies |
| Tests DTO mapping correctness | Yes (unit) | Pure transformation logic |
| Tests boundary values in service logic | Yes (unit) | Can mock dependencies |
| Verifies error response JSON structure | No (E2E) | Requires full stack |
| Verifies database state after operation | No (E2E) | Requires real DB |
| Tests missing auth header returns 401 | No (E2E) | Requires HTTP layer |
| Tests pagination query parameters | No (E2E) | Requires HTTP + DB |

### Step 5: Write test-tracking.json

After generating all test cases and classifying them, write `test-tracking.json` to the same directory as the `functional-spec.md`.

#### test-tracking.json Schema

```json
{
  "itemKey": "{endpoint-id}-{service-name-kebab-case}",
  "itemType": "api-endpoint",
  "generatedAt": "{ISO-8601 timestamp}",
  "functionalSpecPath": "{relative-path-to-functional-spec.md}",
  "testCases": [
    {
      "id": "API-{EndpointID}-{Abbrev}-{Sequence}",
      "name": "{Test case name}",
      "testType": "unit",
      "category": "{category-kebab-case}",
      "priority": "Critical | High | Medium | Low",
      "status": "pending",
      "sourceScenario": "Scenario: {original Gherkin scenario name}",
      "lastError": null,
      "filePath": null,
      "attempts": 0
    }
  ],
  "unitSummary": {
    "total": 0,
    "pending": 0,
    "passing": 0,
    "failing": 0
  },
  "e2eSummary": {
    "total": 0,
    "pending": 0,
    "passing": 0,
    "failing": 0
  }
}
```

#### Field Descriptions

- **itemKey**: Derived from the folder name (e.g., "2984-spring-companymaintenanceservice-getcompany")
- **itemType**: Always "api-endpoint" for this command
- **generatedAt**: ISO-8601 timestamp of generation time
- **functionalSpecPath**: Relative path from repo root to the functional-spec.md
- **testCases**: Array of all test cases (both from spec and gap-fill)
  - **id**: Same test ID as in api-test-cases.md
  - **name**: Human-readable test name
  - **testType**: Always `"unit"` (E2E cases are added later by `/generate-e2e-api-test-cases`)
  - **category**: Kebab-case category (e.g., "success-path", "validation", "authorization", "edge-case", "error-handling", "gap-fill-authorization", "gap-fill-security", "gap-fill-boundary")
  - **priority**: Same priority as in api-test-cases.md
  - **status**: Always `"pending"` at generation time
  - **sourceScenario**: The original Gherkin scenario name (prefix with "Scenario: "). For gap-fill test cases, use "Gap: {gap description}"
  - **lastError**: Always `null` at generation time
  - **filePath**: Always `null` at generation time (populated later when test is implemented)
  - **attempts**: Always `0` at generation time
- **unitSummary**: Aggregate counts for unit test cases
- **e2eSummary**: Set to all zeros (E2E cases are added later by `/generate-e2e-api-test-cases`)

All `unitSummary` counts should reflect the initial state: `total` = count of unit cases, `pending` = same as total, `passing` = 0, `failing` = 0. `e2eSummary` should be all zeros.

## Test Scenario Coverage Requirements

### CRITICAL: Ensure Complete API Test Coverage

Test cases MUST cover a comprehensive range of scenarios to ensure thorough API testing. For each endpoint, ensure the following scenario types are represented:

### 1. Happy Path Scenarios (Success Cases - 2xx Responses)

These verify the API works correctly under normal, expected conditions:

| Scenario Type                  | HTTP Status    | Description                      | Example                                   |
|--------------------------------|----------------|----------------------------------|-------------------------------------------|
| **Primary Success (GET)**      | 200 OK         | Retrieve existing resource       | GET /companies/123 returns company data   |
| **Primary Success (POST)**     | 201 Created    | Create new resource              | POST /contacts creates new contact        |
| **Primary Success (PUT/PATCH)**| 200 OK         | Update existing resource         | PUT /companies/123 updates company        |
| **Primary Success (DELETE)**   | 204 No Content | Delete existing resource         | DELETE /contacts/456 removes contact      |
| **Empty Result Set**           | 200 OK         | Valid query with no matches      | GET /addresses?companyId=999 returns []   |
| **Minimal Input**              | 2xx            | Success with only required fields| POST with only mandatory fields           |
| **Maximum Input**              | 2xx            | Success with all fields populated| POST with all optional fields             |
| **Pagination**                 | 200 OK         | Paginated results                | GET /items?page=2&size=10                 |
| **Filtering**                  | 200 OK         | Filtered results                 | GET /items?status=active                  |
| **Sorting**                    | 200 OK         | Sorted results                   | GET /items?sort=name,asc                  |

### 2. Sad Path Scenarios (Client Error - 4xx Responses)

These verify the API handles invalid requests gracefully:

| Scenario Type              | HTTP Status     | Description                    | Example                                |
|----------------------------|-----------------|--------------------------------|----------------------------------------|
| **Missing Required Field** | 400 Bad Request | Required field not provided    | POST without companyId                 |
| **Invalid Format**         | 400 Bad Request | Field in wrong format          | Invalid date format, malformed JSON    |
| **Invalid Value**          | 400 Bad Request | Value outside valid range      | Negative ID, future date when not allowed |
| **Invalid Data Type**      | 400 Bad Request | Wrong type for field           | String where number expected           |
| **Duplicate Resource**     | 409 Conflict    | Creating duplicate unique record | POST with existing unique key        |
| **Invalid JSON**           | 400 Bad Request | Malformed request body         | Syntax error in JSON                   |
| **Missing Request Body**   | 400 Bad Request | POST/PUT without body          | Empty body on creation                 |
| **Referential Integrity**  | 400/404         | Invalid foreign key            | Company ID doesn't exist               |

### 3. Authorization & Security Scenarios (401/403 Responses)

These verify access control and security requirements:

| Scenario Type                | HTTP Status           | Description                        | Example                             |
|------------------------------|-----------------------|------------------------------------|-------------------------------------|
| **No Authentication**        | 401 Unauthorized      | Missing auth token/header          | Request without Authorization header |
| **Invalid Token**            | 401 Unauthorized      | Malformed or expired token         | Expired JWT token                   |
| **Insufficient Permissions** | 403 Forbidden         | User lacks required role           | Read-only user attempts POST        |
| **Resource-Level Access**    | 403 Forbidden         | User can't access specific resource| User A accessing User B's data      |
| **Rate Limiting**            | 429 Too Many Requests | Exceeded request limits            | Too many requests in time window    |

### 4. Resource Not Found Scenarios (404 Responses)

These verify handling of non-existent resources:

| Scenario Type          | HTTP Status   | Description                      | Example                              |
|------------------------|---------------|----------------------------------|--------------------------------------|
| **Resource Not Found** | 404 Not Found | ID doesn't exist                 | GET /companies/99999                 |
| **Deleted Resource**   | 404 Not Found | Previously existing resource     | GET after DELETE                     |
| **Invalid ID Format**  | 400/404       | ID in wrong format               | GET /companies/abc (expects number)  |
| **Parent Not Found**   | 404 Not Found | Parent resource doesn't exist    | GET /companies/99999/contacts        |

### 5. Boundary & Edge Case Scenarios

These test the limits and edge conditions of the API:

| Scenario Type            | Description                  | Example                            |
|--------------------------|------------------------------|------------------------------------|
| **Minimum Boundary**     | At minimum acceptable value  | Field with exactly 1 character     |
| **Maximum Boundary**     | At maximum acceptable value  | Field at max length (255 chars)    |
| **Just Below Minimum**   | One unit below minimum       | Empty string when min length is 1  |
| **Just Above Maximum**   | One unit above maximum       | 256 chars when max is 255          |
| **Empty String vs Null** | Distinguish empty from null  | "" vs null in optional fields      |
| **Special Characters**   | Unicode, emojis, encoding    | Name with apostrophe, unicode      |
| **Large Payload**        | Maximum request size         | Request body at size limit         |
| **Large Response**       | Large data retrieval         | Query returning 10,000 records     |
| **SQL Injection**        | Security boundary            | Malicious input in parameters      |
| **XSS Attempt**          | Security boundary            | Script tags in input               |

### 6. State Transition Scenarios

These verify correct behavior when entities change state via API:

| Scenario Type          | Description              | Example                              |
|------------------------|--------------------------|--------------------------------------|
| **Valid Transition**   | Allowed state changes    | PATCH /orders/123 status: "shipped"  |
| **Invalid Transition** | Disallowed state changes | "delivered" -> "pending"             |
| **Terminal State**     | Actions on final states  | Attempt to modify "cancelled" order  |
| **Optimistic Locking** | Concurrent modifications | Update with stale version            |

### 7. Temporal Scenarios

These test date/time-related API functionality:

| Scenario Type            | Description                  | Example                  |
|--------------------------|------------------------------|--------------------------|
| **Current Date**         | Operations with today's date | effectiveDate = today    |
| **Future Date**          | Operations with future dates | startDate = next week    |
| **Past Date Rejection**  | Reject historical dates      | Backdate not allowed     |
| **Date Range Validation**| Start before end             | endDate > startDate      |
| **Timezone Handling**    | Cross-timezone operations    | UTC vs local time        |

### 8. Database Verification Scenarios

These verify data persistence after API operations:

| Scenario Type            | Description               | Example                        |
|--------------------------|---------------------------|--------------------------------|
| **Create Verification**  | Data persisted after POST | SELECT confirms new record     |
| **Update Verification**  | Changes persisted after PUT | SELECT confirms updated fields |
| **Delete Verification**  | Data removed after DELETE | SELECT returns no rows         |
| **Cascade Verification** | Related data updated      | Child records updated/deleted  |
| **Audit Trail**          | Changes logged            | Audit table has new entry      |
| **Transaction Rollback** | Partial failure cleanup   | No orphan records on error     |

### 9. Error Response Format Scenarios

These verify consistent error response structure:

| Scenario Type            | Description               | Example                            |
|--------------------------|---------------------------|------------------------------------|
| **Error Body Structure** | RFC 7807 Problem Details  | type, title, status, detail fields |
| **Field-Level Errors**   | Validation error details  | errors array with field names      |
| **Error Message Clarity**| User-friendly messages    | Clear, actionable error text       |
| **No Sensitive Data**    | Security in errors        | No stack traces, SQL, paths        |

### 10. Performance & Reliability Scenarios

These verify API behavior under stress:

| Scenario Type          | Description            | Example               |
|------------------------|------------------------|-----------------------|
| **Response Time**      | Within SLA             | Response < 200ms      |
| **Timeout Handling**   | Long-running operation | 504 Gateway Timeout   |
| **Retry Behavior**     | Idempotent operations  | Same result on retry  |
| **Concurrent Requests**| Race conditions        | Simultaneous updates  |
