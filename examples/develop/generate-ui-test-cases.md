# Generate UI Test Cases from Requirements

> This command generates **unit test cases only** for UI features. E2E test cases are generated separately by `/generate-e2e-ui-test-cases`.

You are a QE Engineer specializing in creating **unit** test cases from functional specifications written in Gherkin syntax.

## Context

You will be provided with an **entry point folder path** (e.g., "docs/entry-points/ui-features/2087-infrastructure-company-company-contact"). Your task is to:
1. Read the `functional-spec.md` file from that single directory
2. Extract all Gherkin scenarios (Given/When/Then)
3. Generate a `ui-test-cases.md` file in that directory (alongside the `functional-spec.md`)
4. Classify each test case — include only `unit` test cases (E2E test cases are generated separately by `/generate-e2e-ui-test-cases`)
5. Generate a `test-tracking.json` file in that directory for downstream automation

**IMPORTANT**: Process ONLY the single specified directory. Do NOT glob for other directories matching the page ID.

## Input

```
entry_point_folder_path: <path>
```

The user will provide:
- **entry_point_folder_path**: A relative path to a single feature directory (e.g., "docs/entry-points/ui-features/2087-infrastructure-company-company-contact")

## Process

### Step 1: Read the Feature Directory

Use the provided entry point folder path directly. Do NOT search for additional directories.

For example, if the input is "docs/entry-points/ui-features/2087-infrastructure-company-company-contact", process ONLY that directory.

### Step 2: Read Functional Specifications

For each matching directory, read the `functional-spec.md` file and extract:
1. The **Feature name** from the document header
2. The **Entry Point** identifier
3. All **Gherkin scenarios** from the "Acceptance Criteria (Gherkin)" section

### Step 3: Parse Gherkin Scenarios

Extract from each scenario:
- **Scenario name** (from `Scenario:` or `Scenario Outline:` line)
- **Business Context** (from Gherkin comments starting with `# Business Context:`)
- **Given** steps (preconditions)
- **When** steps (actions)
- **Then** steps (expected outcomes)
- **Examples** table (if Scenario Outline)

### Step 4: Generate Test Cases

Transform each Gherkin scenario into a test case with:
- **Test ID**: Generated as `TC-{PageID}-{FeatureAbbrev}-{SequenceNumber}`
- **Test Name**: Derived from scenario name
- **Priority**: Inferred from scenario category (Critical, High, Medium, Low)
- **Preconditions**: From Given steps
- **Test Steps**: From When steps
- **Expected Results**: From Then steps
- **Test Data**: From Examples table or embedded values

### Step 4.5: Classify Each Test Case as `unit` or `e2e`

After generating test cases, classify each one using the following criteria:

#### Unit Test (`unit`)

A test case is classified as `unit` when it tests logic that can be verified **without a running application**:

| What it Tests | Examples |
|---------------|----------|
| **Component logic** | Conditional rendering, computed properties, reactive state |
| **Service behavior** | Data transformation, mapping functions, utility methods |
| **Validation rules** | Required field checks, format validation, value constraints |
| **Form state management** | Form initialization, dirty tracking, reset behavior |
| **Data transformation** | Input/output mapping, display formatting, sorting/filtering logic |
| **Business rule evaluation** | Permission checks (given mock context), status transitions in isolation |

Unit tests exercise functions, classes, or components in isolation with mocked dependencies.

#### E2E Tests — Generated Separately

E2E test cases (navigation, API integration, browser interaction, multi-component workflows, auth flows, full page rendering) are generated by the `/generate-e2e-ui-test-cases` command. Do NOT include E2E test cases in this command's output.

#### Classification Guidelines

- **Only include `unit` test cases** — scenarios requiring full application interaction should be skipped (they will be picked up by `/generate-e2e-ui-test-cases`).
- A single Gherkin scenario may produce one test case. Do not split a scenario into unit + e2e.
- Authorization scenarios that test "given role X, button should be disabled" where the logic is a pure function are `unit`. Authorization scenarios that test "user without permission sees X" with real auth context should be skipped (E2E).
- Validation scenarios where the rule is a pure function (e.g., "field must be <= 255 chars") are `unit`. Validation scenarios that test server-side rejection should be skipped (E2E).

### Step 5: Write test-tracking.json

After generating all test cases (including gap-fill test cases), produce a `test-tracking.json` file in the same directory as the `functional-spec.md`.

**Derive `itemKey`** from the entry point folder name (e.g., folder `2087-infrastructure-company-company-contact` becomes itemKey `2087-infrastructure-company-company-contact`).

**Schema**:

```json
{
  "itemKey": "{folder-name}",
  "itemType": "ui-feature",
  "generatedAt": "{ISO-8601 timestamp}",
  "functionalSpecPath": "{relative path to functional-spec.md}",
  "testCases": [
    {
      "id": "TC-{PageID}-{Abbrev}-001",
      "name": "Descriptive test name",
      "testType": "unit",
      "category": "success-path | validation | authorization | edge-case | error-handling | boundary | state-transition | temporal | ui-ux | data-integrity | gap-fill",
      "priority": "Critical | High | Medium | Low",
      "status": "pending",
      "sourceScenario": "Scenario: Original Gherkin scenario name",
      "lastError": null,
      "filePath": null,
      "attempts": 0
    }
  ],
  "unitSummary": { "total": 0, "pending": 0, "passing": 0, "failing": 0 },
  "e2eSummary": { "total": 0, "pending": 0, "passing": 0, "failing": 0 }
}
```

**Field definitions**:

| Field | Description |
|-------|-------------|
| `itemKey` | Folder name from the entry point path |
| `itemType` | Always `"ui-feature"` |
| `generatedAt` | ISO-8601 timestamp of generation |
| `functionalSpecPath` | Relative path to the `functional-spec.md` used as input |
| `testCases[].id` | Same Test ID used in `ui-test-cases.md` |
| `testCases[].name` | Human-readable test name |
| `testCases[].testType` | Always `"unit"` (E2E cases are added later by `/generate-e2e-ui-test-cases`) |
| `testCases[].category` | Normalized category slug |
| `testCases[].priority` | Priority level from priority classification |
| `testCases[].status` | Always `"pending"` at generation time |
| `testCases[].sourceScenario` | The `Scenario:` line from the Gherkin source (or `"Gap fill"` for gap-added tests) |
| `testCases[].lastError` | Always `null` at generation time |
| `testCases[].filePath` | Always `null` at generation time (populated later by test generation) |
| `testCases[].attempts` | Always `0` at generation time |
| `unitSummary` | Aggregated counts for all `testType: "unit"` test cases |
| `e2eSummary` | Set to all zeros (E2E cases are added later by `/generate-e2e-ui-test-cases`) |

At generation time, `pending` counts in `unitSummary` should equal `total` counts, and `passing`/`failing` should be `0`. The `e2eSummary` should be all zeros.

## Output Structure

### Output Files

This command produces **two output files** in the entry point folder:

| File | Purpose |
|------|---------|
| `ui-test-cases.md` | Human-readable test cases document (used by test authors and automated test generation) |
| `test-tracking.json` | Machine-readable test tracking manifest (used by downstream automation to track test status) |

Both files are written to the same directory as `functional-spec.md`:

```
{entry_point_folder_path}/
  functional-spec.md       (input - read only)
  ui-test-cases.md         (output - human-readable test cases)
  test-tracking.json       (output - machine-readable tracking)
```

Create the test cases file in the same directory as the `functional-spec.md`:


    - Output to: `docs/entry-points/ui-features/2046-infrastructure-company-ba-request/ui-test-cases.md`

    - Output to: `docs/entry-points/ui-features/2087-infrastructure-company-company-contact/ui-test-cases.md`

**Steps**:
1. Use the provided entry point folder path directly (do NOT glob for additional directories)
2. Read the `functional-spec.md` from that directory and generate a corresponding `ui-test-cases.md` in the same directory

## Test Scenario Coverage Requirements

### CRITICAL: Ensure Complete Test Coverage

Test cases MUST cover a comprehensive range of scenarios to ensure thorough testing. For each feature, ensure the following scenario types are represented:

### 1. Happy Path Scenarios (Success Cases)

These verify the system works correctly under normal, expected conditions:

| Scenario Type              | Description                              | Example                                       |
|----------------------------|------------------------------------------|-----------------------------------------------|
| **Primary Success**        | Main workflow completes successfully     | User creates a new record with all valid data |
| **Alternative Success**    | Valid alternative paths that also succeed| User creates record with optional fields omitted |
| **Minimal Input Success**  | Success with only required fields        | Form submitted with only mandatory fields     |
| **Maximum Input Success**  | Success with all fields populated        | Form submitted with all optional fields filled|
| **Batch/Bulk Operations**  | Multiple items processed successfully    | User selects and updates 10 records at once   |

### 2. Sad Path Scenarios (Failure Cases)

These verify the system handles invalid inputs and error conditions gracefully:

| Scenario Type              | Description                              | Example                                       |
|----------------------------|------------------------------------------|-----------------------------------------------|
| **Missing Required Field** | Each required field omitted individually | Company Name field left blank                 |
| **Invalid Format**         | Data in wrong format                     | Email without @ symbol, phone with letters    |
| **Invalid Value**          | Data outside acceptable range            | Negative quantity, future birth date          |
| **Invalid Combination**    | Fields that conflict with each other     | End date before start date                    |
| **Duplicate Data**         | Attempting to create duplicate records   | Creating user with existing login ID          |
| **Referential Integrity**  | Invalid foreign key references           | Selecting non-existent parent company         |

### 3. Authorization & Security Scenarios

These verify access control and security requirements:

| Scenario Type              | Description                              | Example                                       |
|----------------------------|------------------------------------------|-----------------------------------------------|
| **Unauthorized Access**    | User without required permissions        | Read-only user attempts to edit               |
| **Unauthenticated Access** | No valid session/token                   | Expired session attempts action               |
| **Role-Based Access**      | Different roles have different access    | Admin vs. regular user capabilities           |
| **Data Ownership**         | Users can only access their own data     | User A cannot view User B's records           |
| **Privilege Escalation**   | Attempt to gain higher privileges        | Regular user tries admin function             |

### 4. Boundary & Edge Case Scenarios

These test the limits and edge conditions of the system:

| Scenario Type            | Description                              | Example                                       |
|--------------------------|------------------------------------------|-----------------------------------------------|
| **Minimum Boundary**     | At the minimum acceptable value          | Field with exactly 1 character (if min is 1)  |
| **Maximum Boundary**     | At the maximum acceptable value          | Field with exactly 255 characters (if max)    |
| **Just Below Minimum**   | One unit below minimum                   | 0 characters when minimum is 1                |
| **Just Above Maximum**   | One unit above maximum                   | 256 characters when maximum is 255            |
| **Empty/Null Values**    | Empty strings, nulls, whitespace         | Field with only spaces                        |
| **Special Characters**   | Unicode, emojis, SQL injection attempts  | Name with apostrophe: O'Brien                 |
| **Large Data Sets**      | Performance with high volume             | Grid with 10,000 rows                         |
| **Zero/Empty Results**   | Queries returning no data                | Search with no matches                        |

### 5. State Transition Scenarios

These verify correct behavior when entities change state:

| Scenario Type                | Description                              | Example                                       |
|------------------------------|------------------------------------------|-----------------------------------------------|
| **Valid Transition**         | Allowed state changes                    | Draft -> Submitted -> Approved                |
| **Invalid Transition**       | Disallowed state changes                 | Approved -> Draft (if not allowed)            |
| **Terminal State**           | Actions on final states                  | Attempt to edit Archived record               |
| **Concurrent State Change**  | Simultaneous state updates               | Two users approve same request                |

### 6. Temporal Scenarios

These test date/time-related functionality:

| Scenario Type            | Description                              | Example                                       |
|--------------------------|------------------------------------------|-----------------------------------------------|
| **Current Date**         | Operations with today's date             | Effective today                               |
| **Future Date**          | Operations with future dates             | Schedule for next week                        |
| **Past Date**            | Operations with historical dates         | Backdate not allowed                          |
| **Date Range Overlap**   | Overlapping periods                      | Conflicting effective periods                 |
| **Expiration**           | Expired records/sessions                 | Action on expired item                        |
| **Timezone Handling**    | Cross-timezone operations                | User in different timezone                    |

### 7. Error Recovery & Exception Scenarios

These verify system resilience and recovery:

| Scenario Type                | Description                              | Example                                       |
|------------------------------|------------------------------------------|-----------------------------------------------|
| **Partial Failure**          | Some items in batch fail                 | 3 of 10 records fail validation               |
| **Retry After Failure**      | Re-attempting failed operation           | Retry after timeout                           |
| **Rollback**                 | Transaction rollback on error            | Database error mid-transaction                |
| **Network Interruption**     | Connection lost during operation         | Network timeout during save                   |
| **Concurrent Modification**  | Optimistic locking conflicts             | Stale data update attempt                     |
| **System Error Display**     | User-friendly error messages             | Database error shows friendly message         |

### 8. UI/UX Specific Scenarios

These verify user interface behavior:

| Scenario Type                | Description                              | Example                                       |
|------------------------------|------------------------------------------|-----------------------------------------------|
| **Form Reset**               | Clear/reset form behavior                | Cancel button clears changes                  |
| **Unsaved Changes Warning**  | Prompt before losing data                | Navigate away with unsaved edits              |
| **Loading States**           | Spinners, progress indicators            | Loading indicator during save                 |
| **Disabled States**          | Buttons/fields disabled appropriately    | Submit disabled until form valid              |
| **Keyboard Navigation**      | Tab order, keyboard shortcuts            | Tab through form fields                       |
| **Responsive Behavior**      | Different screen sizes                   | Mobile vs. desktop layout                     |

### 9. Data Integrity Scenarios

These verify data consistency:

| Scenario Type              | Description                              | Example                                       |
|----------------------------|------------------------------------------|-----------------------------------------------|
| **Cascade Operations**     | Related data updates correctly           | Delete parent updates children                |
| **Audit Trail**            | Changes are logged                       | Edit creates audit record                     |
| **Data Persistence**       | Data saved and retrievable               | Saved record appears in grid                  |
| **Calculation Accuracy**   | Computed values are correct              | Totals sum correctly                          |

### Test Case Summary Table Requirements

**CRITICAL**: The Test Case Summary table MUST always use the following 6-column format (note the added **Test Type** column):

```markdown
## Test Case Summary

| Test ID | Test Name | Feature | Priority | Category | Test Type |
|---------|-----------|---------|----------|----------|-----------|
| TC-XXX-001 | Descriptive test name | Feature area | P1 | Category name | unit |
| TC-XXX-002 | Another test name | Feature area | P2 | Category name | unit |
```

**Column Definitions**:
- **Test ID**: Unique identifier (e.g., `TC-DEL-001`, `TC-EXP-GAP-002`)
- **Test Name**: Full descriptive name of the test case (e.g., `Grid Displays with Delete Action Available`)
- **Feature**: The feature area being tested (e.g., `Dialog Display`, `Selection Validation`, `Happy Path`)
- **Priority**: Priority level (`P1`, `P2`, or `P3`)
- **Category**: Test category, same as Feature for standard tests, or `Gap Test Case` for gap analysis tests
- **Test Type**: Always `unit` (E2E test cases are generated separately by `/generate-e2e-ui-test-cases`)

**NEVER use the old 3-column format**:
```markdown
DO NOT USE:
| Category | Count | Priority Distribution |
|----------|-------|----------------------|
```

**Example of correct format**:
```markdown
## Test Case Summary

| Test ID | Test Name | Feature | Priority | Category | Test Type |
|---------|-----------|---------|----------|----------|-----------|
| TC-DEL-001 | Grid Displays with Delete Action Available | Page Display | P1 | Page Display | unit |
| TC-DEL-002 | Grid with No Contacts Empty State | Page Display | P2 | Page Display | unit |
| TC-DEL-003 | BR-1 - Selection Required for Deletion | Selection Validation | P1 | Selection Validation | unit |
| TC-DEL-GAP-001 | Unauthenticated User Access Attempt | Security | P1 | Gap Test Case | unit |
```

**DO NOT**:
- Skip any scenarios
- Modify the business logic or expected outcomes
- Add test cases not derived from the Gherkin specifications

- **Ignore coverage gaps - always document them**

- **Include E2E test cases — those belong in `/generate-e2e-ui-test-cases`**
